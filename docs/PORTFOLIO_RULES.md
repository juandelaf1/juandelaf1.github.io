# Portfolio Rules

These are immutable rules. Every decision, every component, every page must comply.

---

## Rule 1: Every Project Must Solve a Real Business Problem

No project exists without a clearly stated business context. If the problem is not obvious, the project does not belong.

## Rule 2: Every Project Must Include

- Context: business problem, stakeholders, constraints
- Architecture: how data flows through the system
- Results: specific, measured impact
- Lessons Learned: what would be done differently
- Repository: link to source code
- Future Improvements: what comes next

## Rule 3: No Placeholder Content

Every piece of text must be final. No "Lorem ipsum." No "Coming soon." No "Under construction." If a page is not ready, it does not exist.

## Rule 4: No Unnecessary Animations

Zero animations by default. Transitions only for functional feedback (hover states, link color changes). No fade-ins, no slide-ups, no particle effects, no confetti.

## Rule 5: No Fake Metrics

Every number must be real and verifiable. If a metric cannot be backed by data, it must not appear.

## Rule 6: No Filler Sections

Every section must serve a clear purpose. If a section can be removed without losing meaning, remove it. No "About Me" with personal biography. No "Testimonials." No "Fun Facts."

## Rule 7: No Tutorial Content

This is not a teaching site. Engineering notes are technical documentation, not tutorials. Assume the reader is an experienced engineer. Explain what was done and why, not how to do it step by step.

## Rule 8: Projects Are Case Studies, Not Descriptions

Every project is a structured case study with: problem → challenges → solution → results → lessons. No bullet-point lists of features.

## Rule 9: Technology Is Context, Not the Hero

Describe the tech stack briefly. The focus is on architectural decisions, not which libraries were used. No badge walls of technologies.

## Rule 10: Results Must Be Quantified

Every project must include at least one specific, measured result. Examples: revenue impact, cost reduction, data volume processed, accuracy metric, time saved.

## Rule 11: Homepage Shows Only 4 Projects

The homepage displays exactly four selected projects — the most impressive, most relevant, most complete. No "View all projects" link on the homepage. The projects section is discoverable via navigation.

## Rule 12: Architecture Diagrams Required

Every case study must include an architecture diagram showing how data flows through the system. Text-only architecture descriptions are not sufficient.

## Rule 13: Dark Mode Only

No light mode toggle. The site exists only in dark mode. This is intentional: it signals a developer/technical audience focus.

## Rule 14: No External Dependencies for Content

No CMS. No database. No external API calls at runtime (except build-time GitHub API). All content lives in the repository as Markdown/MDX files.

## Rule 15: No Social Media Integration

No Twitter feed. No Instagram. No Dribbble. Only professional links: LinkedIn, GitHub, Email, Resume.

## Rule 16: The Resume Is a PDF, Not a Page

The resume is a downloadable PDF. Not a recreated HTML version. Not a separate page.
